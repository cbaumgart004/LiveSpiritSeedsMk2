The owner asked whether `/admin` supports drag-and-drop for photos and opacity while iterating on the
home splash. Partly — see below. Filing the gaps so the composition doesn't stay code-only.

**What TinaCMS already gives us**
- **Drag-and-drop upload.** The `image` field opens the media manager, which accepts files dropped
  onto it. So replacing a photo is already a drag-and-drop job, no code.
- **Drag-to-reorder.** `blocks[]` list items can be dragged to reorder on the page, which is what
  re-alternates `imageSide: auto`.

**What it does NOT give us (the actual asks)**
1. **Drag-positioning an element on the page.** There is no canvas-style "drag the bowls where you
   want them". The bowls/flower offsets are CSS (`.tagline__bowls`, `.tagline__flower` in
   `layout.css`), so moving them is a code change today. Options, cheapest first:
   - expose X/Y/size as Tina number fields on the splash block and feed them to CSS custom
     properties (same trick `imageWidth` → `--media-basis` already uses);
   - a custom Tina field component with a small drag-pad that writes those numbers.
2. **An opacity slider.** "Tagline Blend %" is `type: 'number'`, so it renders as a plain number box —
   you type 88, you don't drag it. Tina supports custom field components and this repo already ships
   one (`PreviewLinkField` in `tina/config.ts`), so a range-input slider with a live preview is a
   small, self-contained job.

**Why it matters:** every visual tweak in this area currently costs a round-trip through code. The
owner is actively iterating on the hero, which is exactly the workflow that should be owner-editable
per the project's "content, not code, drives the look" convention (DESIGN.md §7).

**Where things live**
- Fields: `tina/config.ts` (`taglinePlacement`, `taglineBlend` on the `contentSection` template).
- Layers + offsets: `src/styles/layout.css` (`.tagline*`, `.splash--over`).
- Component: `src/components/TaglineArt.jsx`.
- Precedent for a custom field component: `PreviewLinkField`, `tina/config.ts`.
